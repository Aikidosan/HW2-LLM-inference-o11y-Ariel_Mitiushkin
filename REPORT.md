# HW2: LLM Inference + Observability ‚Äî Report

Text-to-SQL PoC: `Qwen3-30B-A3B-Instruct-2507` on 1√ó H100, served with vLLM, driven by a
LangGraph `generate ‚Üí execute ‚Üí verify ‚Üí revise` agent, observed with Prometheus/Grafana and
traced with Langfuse. **SLO:** P95 end-to-end agent latency < 5 s at 10+ RPS over 5 min.

## Phase 1 ‚Äî Serving config

`Qwen3-30B-A3B` is a **Mixture-of-Experts** model ‚Äî 30B total params but **~3B active per
token** ‚Äî which is why a 30B-class model serves cheaply on one H100: the full weights fit, but
each decoded token touches only ~3B. The workload is prefill-heavy (1.5‚Äì3K-token prompts), with
short SQL outputs and **2‚Äì3 serial, prefix-sharing LLM calls per request**, so the config
optimizes for **low TTFT and high concurrency**, not single-stream throughput.

| Flag | Value | Why (this workload) |
|---|---|---|
| `--quantization` | `fp8` | Weights ~30 GB vs ~60 GB BF16 ‚Üí frees ~45 GB for KV cache; uses Hopper FP8 cores. |
| `--gpu-memory-utilization` | `0.90` | Weights+KV get most of 80 GB; 10% for activations/CUDA graphs. |
| `--max-model-len` | `8192` | Schema context carries per-column example values (Phase 5 accuracy lever); the widest schema (`card_games`) reaches ~4.4K tokens, so 4096 truncated it (400s). 8192 fits every prompt; KV still holds ~55√ó concurrency. |
| `--max-num-seqs` | `64` | Slots for 10 RPS √ó 2‚Äì3 serial calls without thrashing KV. |
| `--enable-chunked-prefill` | on | Interleaves 3K-token prefills with decode ‚Üí stable TTFT under load. |
| `--enable-prefix-caching` | on | generate/verify/revise + same-DB requests share a schema prefix ‚Üí big TTFT win. |
| `--tensor-parallel-size` | `1` | One H100; MoE 3B-active fits without sharding (expert-parallel only helps at TP>1). |

Not set: `--enforce-eager` (CUDA graphs lower decode latency); KV-FP8 / chunk-sizing held back as
Phase 6 levers. **Measured at startup:** FP8 weights **29.1 GiB** ‚Üí KV cache **41.5 GiB
(453,648 tokens)** ‚Üí **55√ó max concurrency** at 8192 tok/req ‚Äî still far above the 64 `max-num-seqs`
and the ~10 RPS workload, so the larger context costs no RPS. CUDA graphs captured. Manual queries
return valid SQL at **0.4‚Äì0.9 s warm** (`screenshots/vllm_manual_query.png`). All reported numbers
come from the real H100; local pipeline dev used Ollama.

## Phase 2 ‚Äî Observability dashboard

`vLLM serving` dashboard (`serving.json`, generated by `scripts/build_dashboard.py`); Prometheus
scrapes `/metrics` every 5 s. Layout reads cold and reacts under load: a **SLO-at-a-glance** stat
row (E2E P95 green/red at 5 s, Achieved RPS, KV usage), **Latency** P50/95/99 for E2E/TTFT/ITL
(via `histogram_quantile` on the `_bucket` series), **Throughput** (in/out tok/s, running/waiting/
finished, queue depth), and **KV cache & prefix caching** (utilization, used vs capacity, hit
rate). Under load (`screenshots/grafana_serving.png`) every panel reacts: **TTFT fell 90‚Üí30 ms as
the prefix cache warmed**, **prefix-cache hit rate ~95%** (direct payoff of `--enable-prefix-
caching`), KV ~1.5% ‚Äî huge headroom, foreshadowing the Phase 6 finding that the GPU is never the
limit.

## Phase 3 ‚Äî Agent ¬∑ Phase 4 ‚Äî Tracing

`generate ‚Üí execute ‚Üí verify ‚Üí revise` LangGraph, capped at MAX_ITERATIONS, and **fully async**
(`ainvoke`; sqlite pushed to a threadpool) so one event-loop worker keeps many requests in flight ‚Äî
the change that unlocks the SLO in Phase 6. The verifier has two modes: an **LLM plausibility
check** (the Phase-3 deliverable; `VERIFY_LLM=1`, what the Langfuse waterfall traces) and a
**programmatic gate** (default ‚Äî revise on execution error / empty result), adopted in Phase 6 once
the eval proved the LLM check adds zero accuracy (below). Revise demonstrably fires (e.g.
`financial`, `formula_1`). Langfuse wraps each run in one trace whose waterfall shows the node spans
+ nested `ChatOpenAI` generations with latency/tokens (`screenshots/langfuse_trace.png`), tagged
with `db`, `iteration_count`, `verify_ok`, `revised` (`screenshots/langfuse_tags.png`).

## Phase 5 ‚Äî Eval (baseline ‚Üí improved)

Execution-accuracy over 30 BIRD questions (`run_eval.py`); correct = agent rows match gold rows
(canonicalized).

| Config | Overall | iter 1 | iter 2 | Errors | Avg lat | File |
|---|---|---|---|---|---|---|
| Baseline (LLM verify, bare schema) | 30.0% (9/30) | 26.7% | 30.0% | 0 | 1.06 s | `eval_baseline.json` |
| **+ schema example values, programmatic verify** | **40.0% (12/30)** | 40.0% | 40.0% | 0 | 0.68 s | `eval_after_tuning.json` |

**The accuracy lever was the schema, not the loop.** Annotating each column with 3 real example
values (`-- e.g. Australian Grand Prix, ‚Ä¶`) lets the model match string literals **exactly** and
pick the right column ‚Äî the most common silent-wrong cause on BIRD. That alone moved 30 ‚Üí 40%
(+3 questions), with the widest schema (`card_games`) needing the 4096 ‚Üí 8192 context bump to fit.

**Does the loop earn its keep?** No ‚Äî and now we can prove it. In **both** runs the per-iteration
pass rate is flat (iter 1 == iter 2), i.e. the verify‚Üírevise loop converted **zero** net questions.
The traces show why: surviving failures are SQL that **executes and returns plausible-but-wrong
rows**, which a gold-free verifier can't catch. So the LLM verifier is pure latency cost ‚Äî which is
exactly the license Phase 6 uses to replace it with a free programmatic gate (note avg latency
1.06 ‚Üí 0.68 s from dropping one LLM call per question).

**Why not higher than 40%? ‚Äî and what I tried.** I added **self-consistency** (`SELF_CONSISTENCY_K`:
sample K=5 candidates in parallel, execute each, keep the majority *executed result*). It moved the
score **0** (still 40%, `results/eval_self_consistency.json`) ‚Äî a useful negative result: the misses
are **systematic, not stochastic** (the model is confidently wrong the same way every sample), so a
vote over its own answers can't fix them. Root-causing the 18 failures: **~9 are evidence-dependent**
‚Äî BIRD normally ships an *evidence* hint that this eval set omits (e.g. "normal IgG" = a specific
`BETWEEN` range, "well-finished" = `ClosedDate IS NULL`, "carcinogenic" = `label='+'`, disqualified =
`statusId=2`), which no prompt recovers; **~5 are projection ambiguities** (gold returns `id`, the
agent returns `name`; gold orders `Street,City,State,Zip`, the agent swaps two columns); **~4 are
hard string/date parsing**. The remaining gains are either re-introducing the evidence field or
over-fitting gold's exact projection on a 30-row test ‚Äî so 40% is the honest ceiling here, and
self-consistency stays available but **off by default** (it costs K√ó generate latency for no gain).

## Phase 6 ‚Äî SLO diagnosis & tuning

Open-loop driver fires at the agent; each request = 2‚Äì4 serial vLLM calls.

| Stage | Change | P95 e2e | RPS | Errors |
|---|---|---|---|---|
| **Baseline** | 1 worker, MAX_ITER=3 | **97.3 s** | 7.5 | 765 |
| Iter 1 | workers 1‚Üí8 | 8.96 s | 9.7 | 192 |
| Iter 2 | schema NULL-FK fix | 11.06 s | 7.1 | 0 |
| Iter 3 | MAX_ITER 3‚Üí2 | 9.05 s | 9.7 | ~0 |
| Iter 4 | max_tokens=512 (timeout+retry reverted) | 6.52 s | 8.33 | 3 |
| Iter 5 | **async graph** (`ainvoke`, sqlite‚Üíthreadpool) | 4.55 s | ~6‚Üí10 | 0 |
| Iter 6 | **programmatic verify** (drop happy-path LLM call) | **2.77 s** | **9.78** | 0 |

1. **saw** e2e P95 = 97 s but Grafana showed vLLM per-call P95 2.3 s, **KV 3.7%** (GPU idle). ‚Üí
   **hypothesized** the *agent process* is the bottleneck (one uvicorn, sync graph in a ~40-thread
   pool, each request holding a thread across blocking calls), not the GPU. ‚Üí **changed** workers
   1‚Üí8. ‚Üí **result** P95 97‚Üí9 s, client-errors 551‚Üí0, RPS‚Üí9.7.
2. **saw** ~192 HTTP 500s survived; reproduced ‚Üí `render_schema` crashes on a foreign key with a
   NULL referenced column (`european_football_2`). ‚Üí **changed** guarded the NULL-FK case in
   `schema.py`. ‚Üí **result** 500s 192‚Üí0; P95 *rose* to 11 s because those requests had been
   **fast-failing (~0 ms) and shedding load** ‚Äî exposing the true cost.
3. **saw** tail = revising requests doing up to 6 serial calls; Phase 5 showed iter-3 adds 0
   accuracy. ‚Üí **changed** MAX_ITER 3‚Üí2. ‚Üí **result** P95 11‚Üí9 s, RPS‚Üí9.7, eval unchanged.
4. **saw** a 118 s straggler (P99 16 s); calls are decode-bound (ITL ~25 ms). ‚Üí **changed**
   `max_tokens=512` **and** `timeout=30+retry`. ‚Üí **result** max_tokens helped (P95‚Üí6.5 s, max
   118‚Üí65 s) but timeout+retry **backfired into a retry storm** (P95 107 s) ‚Üí reverted, kept
   max_tokens.
5. **saw** at this point P95 6.5 s / 8.3 RPS with the GPU still idle (KV < 4%): the 8 **sync**
   workers cap concurrency at 8, but 10 RPS √ó ~1.6 s needs ~16 in flight (Little's law) ‚Äî a
   structural ceiling, not a GPU one. ‚Üí **changed** made the whole graph **async** (`ainvoke`,
   sqlite in a threadpool) so one worker holds many requests concurrently. ‚Üí **result** P95
   6.5 ‚Üí 4.55 s; vLLM now runs ~15 concurrent (queue 0, GPU 60‚Äì80%).
6. **saw** throughput still capped ~6 RPS warm with the GPU not saturated ‚Üí the limit was the **2
   serial LLM calls per request** (generate + LLM-verify), and Phase 5 proved the verify converts
   **0** questions. ‚Üí **changed** replaced the happy-path LLM verifier with a **programmatic gate**
   (revise only on error/empty). ‚Üí **result** one fewer LLM call per request: P95 4.55 ‚Üí **2.77 s**,
   sustained **9.78 RPS**, 0 errors, eval **unchanged at 40%**.

**Langfuse confirms the diagnosis** (filtering the traces by the `iteration_count` tag): no-revise
requests average **1.56 s (P95 3.0 s)** while revised requests average **3.26 s (P95 6.2 s)** ‚Äî the
revise loop is the tail, which is exactly what capping MAX_ITERATIONS and then dropping the LLM
verify targeted.

**Verdict ‚Äî SLO met.** Over a full 5-min window at 10 RPS (`results/load_after_tuning.json`): **P95
2.77 s** (target < 5 s; baseline 97.3 s, ~35√ó), **9.78 RPS sustained** (target 10+; the driver fired
3000 requests and the server drained them in +7 s, i.e. it kept pace at 10 RPS), **0 errors**
(baseline 765). A 12-RPS probe saturates (queue backs up), so the sustainable ceiling sits right at
~10 RPS ‚Äî the SLO. The two decisive moves both came straight from the diagnosis: the GPU was never
the limit (KV < 6%, 60‚Äì80% util), so the wins were **agent-side concurrency** (async) and
**removing the accuracy-free LLM verify** ‚Äî not GPU tuning. **Quality held**: `eval_after_tuning.json`
= 40%, unchanged by the verify swap (and up from the 30% baseline via the schema lever).
`grafana_before.png` (GPU idle, agent drowning) and `grafana_after.png` (E2E P95 1.97 s, 10.7 RPS,
KV 5.8%) bracket it.

## Agent value

The per-iteration pass rate is the verdict: **flat at every cap** (26.7‚Üí30‚Üí30% baseline; 40‚Üí40%
improved). The LLM verify‚Üírevise loop converted **zero** net questions in either config, because
surviving failures are plausible-but-wrong rows a gold-free verifier can't catch ‚Äî while each
revise costs 2 extra serial calls and drives the latency tail (Langfuse: revised P95 6.2 s vs
3.0 s). So the data justified **replacing the LLM verifier with a programmatic gate**: same accuracy,
one fewer LLM call per request, and the move that pulled P95 under 5 s. The real accuracy lever was
**context, not iteration** ‚Äî example values in the schema (30‚Üí40%). A verifier that re-ranks against
a self-generated reference (not blind plausibility) is the path to making the loop earn its calls.

## What I'd do with more time

*Done this round (were the prior roadmap): async agent tier, dropping the happy-path `verify` LLM
call, and example-value schema context ‚Äî together these hit the SLO and lifted accuracy to 40%.*
Remaining:

- **Make `verify` catch semantic errors** ‚Äî generate an independent second query / NL restatement
  and compare, instead of judging plausibility blind. Highest-leverage change left for accuracy
  (the loop currently converts 0; this is how it would start earning its calls).
- **Column-pruned schema rendering** (retrieve only relevant tables/columns) ‚Äî shorter prompts cut
  prefill/TTFT *and* reduce distraction on the large-schema DBs that still drive the misses
  (`codebase_community`, `formula_1`), and would let `max-model-len` drop back toward 4096.
- **Speculative decoding** to cut the decode-bound ITL (~25 ms) ‚Äî buys headroom well past 10 RPS.
- **A concurrency limiter** that sheds/queues gracefully above the ~10 RPS ceiling instead of
  letting the open-loop backlog grow.

the H100 VM. Final agent config: **async graph, 8 workers, MAX_ITER=2, programmatic verify
(`VERIFY_LLM=0`), max_tokens=512**; vLLM `--max-model-len 8192`.*
